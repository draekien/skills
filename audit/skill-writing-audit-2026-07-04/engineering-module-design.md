# module-design

## Cross-skill script dependency: `query.py` / `write.py` are not bundled with module-design

**Gap:** `SKILL.md` Session Start (step 2, "Load ubiquitous language") and `references/ddd-mode.md` both instruct running `scripts/query.py` and `scripts/write.py` relative to module-design's own skill root. Neither script is bundled in `skills/engineering/module-design/scripts/` (which contains only `skillsrc.py`) — both live in `skills/engineering/get-specific/scripts/`. As written, these invocations will fail with "file not found" whenever DDD mode activates (i.e. whenever `.draekien/ubiquitous-language.yaml` exists). Unlike the `skillsrc.py` step, there is no fallback path documented for `query.py`/`write.py`, so this is a hard failure, not a degraded path.

**Why this is a tradeoff, not a clear-cut fix:** there are three structurally different ways to resolve it, each with real consequences:

1. **Bundle copies of `query.py`/`write.py` into module-design's own `scripts/` directory.** Mirrors the self-contained-script convention already used for `skillsrc.py` (per `specs/skillsrc.md`'s "every skill that reads or writes `.skillsrc` must bundle its own copy" rule), but creates and must maintain two independent copies of the same dictionary read/write logic — a duplication/drift risk if `get-specific`'s dictionary format or CLI ever changes.
2. **Reference the scripts at their real cross-skill path** (e.g. `../get-specific/scripts/query.py`) and document an explicit hard dependency on the `get-specific` skill being installed alongside module-design. This avoids duplication but introduces an inter-skill coupling that doesn't otherwise exist elsewhere in this repo's skill set, and breaks if `get-specific` is renamed, moved, or not installed.
3. **Delete the duplicated DDD-mode term-capture/conflict-detection logic from `ddd-mode.md` and Session Start entirely, and delegate to invoking the `get-specific` skill instead.** This is the least duplicative option, since `get-specific`'s own `SKILL.md` already fully specifies this exact workflow (Term Capture, Conflict Detection, Session Start dictionary loading). But `get-specific` has `disable-model-invocation: true`, so this would change how DDD mode gets triggered (explicit user invocation vs. automatic activation) and removes process logic (the "During Interview" term-capture and conflict-detection steps currently embedded in `ddd-mode.md`) that may be intentionally duplicated for a self-contained DDD Mode experience inside module-design.

**Recommendation:** Option 1 (bundle copies) is closest to the repo's existing convention (each skill self-bundles its `.skillsrc` script rather than reaching into another skill's directory) and keeps module-design usable standalone. It also matches the "Available scripts" listing convention already used for `skillsrc.py`. I'd lean this way, updating the "Available scripts" section to list `scripts/query.py` and `scripts/write.py` alongside `skillsrc.py`.

**Alternatives:** Option 2 (cross-skill path reference) is viable if the repo intends module-design and get-specific to always be installed together — but that dependency should then be stated explicitly in module-design's frontmatter `description` and in `DDD Mode`'s activation section, not left implicit. Option 3 (delegate wholesale to get-specific) is the most architecturally clean but requires deciding whether DDD mode should remain an automatic, in-session behavior of module-design or become an explicit hand-off to a separate skill — a scope-of-skill decision beyond a documentation fix.
