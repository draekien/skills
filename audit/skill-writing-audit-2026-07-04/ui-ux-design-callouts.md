# design-callouts

## Contradiction between the "Never the only channel" commitment and the pre-flight essential-content gate

**Gap:** Line 10 (commitment 2) reads: "Transient and hover-triggered surfaces may amplify essential content but must never own it." This explicitly permits a tooltip or toast to carry essential content, provided it also exists elsewhere ("amplify"). The pre-flight gate at lines 29–30 reads: "Content essential to completing the task → main body, field label, or button label — never a callout," which is an absolute ban with no duplication exception. The Attention & onboarding tooltip rule at line 65 ("Content must pass the tooltip test: not essential to the task, and not already visible") sides with the absolute ban. An agent applying this skill to, say, a tooltip that restates a password's required format (already shown in a persistent hint below the field) cannot determine from the document whether that counts as sanctioned "amplification" or an automatic gate failure.

**Why this is a tradeoff, not a clear-cut fix:** Reconciling the two statements requires deciding actual product-design policy, not just choosing which text is a leftover duplicate:

- If the gate (lines 29-30) is authoritative, the "may amplify" clause in the commitment (line 10) must be deleted or narrowed to something that never actually results in essential content appearing in a callout — which changes what commitment 2 is allowed to justify elsewhere in the skill (e.g., it currently is the doctrinal basis invoked by "Never the only channel").
- If the commitment (line 10) is authoritative, the gate needs a carve-out for content that already has a persistent home — which changes the pre-flight checklist's behavior (an agent would need to check for a persistent duplicate before rejecting a callout, rather than rejecting on essentiality alone) and would need to be reconciled with the tooltip test at line 65 too.
- The docs-admonitions reference elsewhere in the skill family grants a similar sanctioned-duplication exception for hazard warnings (repeating a destructive-step warning immediately before the step), suggesting the intended policy may be the carve-out — but that is an inference, not something stated in this file, and extending it to tooltips/toasts is a scope decision someone with product authority over this skill should confirm.

**Recommendation:** Adopt the carve-out reading — amend the gate at lines 29-30 to read: "Content essential to completing the task, with no persistent copy elsewhere → main body, field label, or button label, never a callout. (A callout may still echo essential content that already has a persistent home — e.g., a warning restating a destructive step immediately before it — but must never be its only copy.)" and soften line 10's "amplify" wording to make explicit that only echoing (never sole-sourcing) is permitted. This keeps continuity with the sanctioned-duplication precedent already used for hazard warnings in the docs-admonitions material.

**Alternative:** Keep the gate absolute and strike "may amplify essential content" from line 10, narrowing commitment 2 to "Transient and hover-triggered surfaces must never carry essential content — omit it, or point to where it lives." This is simpler and removes the contradiction, but forecloses the hazard-warning-style duplication pattern the skill's own admonitions guidance elsewhere seems to rely on, so it may require a follow-up check of `references/docs-admonitions.md` for consistency.
